# BoundaryQAT 实施规划

## 目标体验

完成一个可复现、可解释、可上板的端侧瞳孔分割项目。展示重点是 ESP32-S3 约束下的图像算法工程闭环，而不是未经验证的算法首创声明。

## 总体成功标准

1. FP32 模型可在按受试者或完整序列隔离的固定数据切分上复现分割与中心定位指标，并至少在一个真实采集的公开数据集上完成主评估。
2. 至少一个 INT8 `.espdl` 模型可在 ESP32-S3 上完成端到端推理。
3. README 记录相同测试集上的 FP32、INT8 基线与边界指标驱动配置，并列出实机延迟、RAM 和 Flash。
4. 每个结论均可由配置、随机种子、检查点和结果文件追溯；未验证的假设明确标注为假设，不能宣称算法首创或临床可用。

## 阶段 0：部署可行性探针

### 目标

在投入训练与算法优化前，确认 LightUNet 的导出图、算子与 ESP32-S3 后端兼容。

### 输入与输出

- 输入：当前 LightUNet、固定随机输入与 ESP-DL/ESP-PPQ 环境。
- 输出：ONNX 导出结果、ESP-PPQ 量化日志、算子支持检查结果与最小 `.espdl` 模型。

### 验收

- ONNX 导出成功，且 ONNX Runtime 与 PyTorch 的输出误差在预设容差内。
- ESP-PPQ 可生成目标为 ESP32-S3 的 `.espdl`，或明确记录阻塞算子。

### 失败路径

- 若 `Resize`、`Concat`、`Sigmoid` 或其他算子不兼容，先做最小模型替换，不新增自定义 ESP-DL 算子。
- 若模型内存超过板卡可用空间，先缩小通道数或减少层数，再重新验证。

## 阶段 1：数据与 FP32 基线

具体到数据、处理部件、传输方向和输出的完整说明见[阶段 1：具体数据流程](阶段1_具体数据流程.md)。阶段 1 全部运行在电脑端，不应与阶段 4 的 ESP32-S3 实机图像链路混写。

当前状态：已完成。LPW-only 测试 Dice 为 0.9021、Boundary IoU 为 0.6474、中心误差为 1.467 px；固定口径与限制见[阶段 1：FP32 基线结果](阶段1_FP32基线结果.md)。

### 目标

建立合法、可复现的数据读取、训练、验证和评估流程。

### 输入与输出

- 输入：至少一个真实采集的公开瞳孔分割数据集、可选合成数据集及其许可说明。
- 输出：按受试者或完整序列隔离的数据切分、预处理定义、训练配置、最佳 FP32 检查点、指标 JSON 与预测可视化。
- 当前数据：采用 LPW 原始视频/中心点标注与其第三方 GT 分割掩码；目录、抽样范围、许可边界和引用见[数据来源与许可](数据来源与许可.md)。

### 验收

- 单元测试覆盖数据形状、掩码取值和评价指标。
- 在固定随机种子下可复现训练与验证流程。
- 报告 Dice、Boundary IoU、瞳孔中心 MAE，并将边界带宽、后处理与中心计算方法写入评估文档；边界 Dice、内部 Dice 作为诊断指标。

### 失败路径

- 若 64×64 输入无法稳定定位瞳孔中心，先以更高分辨率完成离线基线，再评估降采样损失。
- 合成数据不能作为唯一测试来源；若公开数据与自采红外图存在明显域差异，自采数据只作为外部测试集，不混入主要结论。

## 阶段 2：ESP32-S3 INT8 基线

当前状态：软件侧基线已完成。已建立独立 64×64 FP32 基线、真实训练样本校准清单、PTQ 同口径评估、层级误差报告及 `.espdl` 导出；测试 Dice 为 FP32 0.8198、PTQ 0.8227。尚缺 ESP32-S3 实机加载验收，详见[阶段 2：64×64 与 PTQ 基线结果](阶段2_64x64与PTQ基线结果.md)。

### 目标

建立与真实后端一致的 PTQ-INT8 基线，先回答“能否部署”和“损失多少”。

### 输入与输出

- 输入：阶段 1 FP32 检查点与校准集。
- 输出：PTQ 模型、量化配置、`.espdl` 文件、同一测试集的量化指标与层级误差分析。

### 验收

- `.espdl` 模型可由 ESP-DL 加载。
- FP32、PTQ 评估严格采用同一测试集与后处理。
- 报告 Dice、Boundary IoU、中心定位指标及诊断指标的变化。

### 失败路径

- 若 PTQ 精度不足，先检查校准集覆盖度、量化日志和层级误差，不直接修改数据切分。
- 若必须改变模型或输入分辨率，建立新的基线编号，不与旧结果混写。

## 阶段 3：边界指标驱动的量化优化

当前状态：软件侧实验已完成。严格 PTQ 与 TQT 候选未能把验证集几何约束稳定带到测试集；逐样本审计确认困难条件覆盖不足后，使用合法 HMEPS 训练数据重建独立 FP32/INT8 基线。最终 INT8 相对阶段 2 标准 INT8 将测试 Boundary IoU 从 0.4007 提升至 0.5028、中心误差从 3.041 px 降至 1.415 px、空预测率从 2.564% 降至 0%。当前候选为 `MinMax + 128 张 LPW 校准样本 + 阈值 0.40`，实机一致性待开发板到达后补验，详见[阶段 3：严格约束优化与辅助数据结果](阶段3_严格约束优化与辅助数据结果.md)。

### 目标

在 ESP32-S3 的逐张量、对称、二的幂量化约束内，比较标准配置与边界导向的离线配置选择。

### 方法边界

- 仅搜索目标后端支持的校准、阈值或 TQT 配置。
- 选择标准预先固定，例如：在整体 Dice 下降不超过预设阈值时，最大化 Boundary IoU。
- 所有选择只在验证集完成；最终测试集只用于一次性报告。
- 不引入输入相关的运行时通道分类、动态 scale 或额外推理分支。
- 仅当训练阶段实际使用伪量化并引入边界损失时，才将对应实验组命名为“Boundary-QAT”；其他离线搜索统一命名为“Boundary-guided tuning”。

### 验收

- 完整保存候选配置、验证集选择过程与最终测试结果。
- 报告改进与代价；若没有改进，也如实保留负结果。
- 验证导出模型在 ESP32-S3 后端保持相同图结构与运行路径。

### 失败路径

- 若边界导向配置无收益，项目仍以“量化误差按区域分析 + 端侧部署闭环”交付。
- 若 TQT 与导出工具不兼容，退回 PTQ 阈值搜索，不伪称 QAT 结果。

## 阶段 4：实机部署与性能测量

当前状态：开发板已到达，但随板摄像头为可见光 OV2640；当前电脑尚未发现可直接调用的 ESP-IDF 命令行环境。阶段 4 将按“工具链与板卡盘点 → 固定图模型加载 → 可见光相机链路 → 后续红外域验收”推进。可见光相机只用于工程闭环，不替代红外眼图精度验证。具体见[阶段 4：ESP32-S3 实机部署实施规划](阶段4_ESP32-S3实机部署实施规划.md)。

### 目标

在 ESP32-S3 上完成相机输入、预处理、模型推理、二值掩码与中心坐标输出。

### 验收

- 同一张测试图在 PC 端量化模拟与板端输出一致或误差可解释。
- 报告预处理、推理与后处理的分段耗时，避免只报模型前向耗时。
- 只有测量了能耗，才报告 mJ/帧；没有测量工具时不估算。

### 失败路径

- 若实时帧率未达预期，优先优化输入与内存布局，再考虑缩小模型。
- 若相机调试阻塞，先以固定图像完成模型上板验证，采集硬件作为后续工作。

## 阶段 5：成果整理与求职呈现

### 输出

- 更新后的中英文 README。
- 一页结果总表、误差热图、模型结构图与实机照片/短视频。
- 可复现命令、环境版本、模型检查点下载说明与数据许可说明。
- 简历项目描述：问题、个人职责、技术路线、量化结果、实机指标与一个真实取舍。

### 验收

- 面试官可在三分钟内理解项目目标、约束、结果与个人贡献。
- 所有数据均为真实测得；计划项、失败项和限制项清楚标示。

## 推荐顺序与时间盒

| 阶段 | 建议投入 | 完成后再进入下一阶段的条件 |
|---|---:|---|
| 阶段 0 | 3—5 天 | `.espdl` 导出可行或已明确最小替代模型 |
| 阶段 1 | 1—2 周 | 有可复现 FP32 指标 |
| 阶段 2 | 3—5 天 | 有可上板 INT8 基线与量化损失数据 |
| 阶段 3 | 1—2 周 | 有完整对照表，无论结果正负 |
| 阶段 4 | 1 周 | 有实机日志与性能数据 |
| 阶段 5 | 2—3 天 | README、图表和简历表述完整 |

秋招时间紧时，阶段 0、1、2、4 是必须完成项；阶段 3 的优化只在基线闭环成功后开展。
