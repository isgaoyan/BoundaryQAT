# 阶段 4：ESP32-S3 实机部署实施规划

## 当前硬件事实

- 开发板已到达，主控仍按 `ESP32-S3-WROOM-1-N16R8` 规划。
- 随板摄像头为可见光 OV2640，不是原计划的红外采集模组。
- 芯片已通过 TTL 串口 `COM5` 实测识别为 ESP32-S3 QFN56 rev0.2，40MHz 晶振、16MB Quad Flash、3.3V Flash 与 8MB 内置 PSRAM，确认满足 N16R8 目标配置。
- 开发板具体商品型号与摄像头引脚表仍待实物资料确认；确认前不锁定相机引脚配置。
- 当前电脑已安装 ESP-IDF `v5.4.2`、esptool `4.12.0` 与 ESP32-S3 工具链，并通过官方 `hello_world` 完整编译验收。
- 开发板 TTL 接口已识别为 `COM5`；原厂 16MB Flash 已备份，ESP-IDF `hello_world` 已按 16MB Flash 配置完成烧录与串口启动验收。

## 目标体验

先证明锁定的 INT8 模型能够在 ESP32-S3 上加载、执行并输出可比对的张量，再接入 OV2640 完成实时采集链路。可见光画面仅用于工程调试，不用于替代红外眼图评价模型精度。

## 固定部署候选

- 模型：`results/runs/stage3_lpw_hmeps_ptq_test_v1/light_unet_64_boundary_guided_esp32s3.espdl`。
- 输入：单张、单通道、`64×64`。
- 后处理：统一阈值 `0.40`、最大连通区域、质心坐标。
- 量化工具：ESP-PPQ `1.3.6`。
- 部署框架：与当前 `.espdl` 格式兼容的 ESP-DL 3.x；ESP-IDF 使用 ESP-DL 官方要求的 `release/v5.3` 或更高兼容版本，首次成功后固定精确版本。

不得根据板端结果重新查看测试集并调整模型、校准集或阈值；若模型格式不兼容，先修正工具版本配对并建立新的导出编号。

## 分步实施与验收

### 4A：硬件与工具链盘点

当前状态：已完成。正式工程 `deployment/esp32s3` 已通过 `COM5` 完成构建、烧录和串口验收。启动日志确认 ESP-IDF `v5.4.2`、ESP32-S3 rev0.2、16MB DIO Flash、8MB 80MHz Octal PSRAM；PSRAM 启动内存测试通过，可用约 8189KB。未发现崩溃、看门狗复位或内存初始化错误。

输入：开发板正反面型号、模组丝印、摄像头排线/引脚资料、数据 USB 线。

输出：板卡型号、芯片/Flash/PSRAM、摄像头引脚、串口号、ESP-IDF 与 ESP-DL 精确版本记录。

验收：最小固件可编译、烧录并通过串口打印芯片、Flash 和 PSRAM 信息。

失败路径：若无串口，先排查数据线、USB 驱动、BOOT/RESET 操作与供电；不在型号未确认时套用其他板卡的摄像头引脚。

### 4B：固定图模型加载

输入：锁定 `.espdl`、一张来自固定测试集的 `64×64` 灰度输入及 PC 量化模拟参考输出。

处理：模型从 Flash 加载；固定输入进入 ESP-DL；串口输出模型加载状态、输出张量形状、校验摘要和分段耗时。

输出：板端原始输出或足以逐项核对的完整输出摘要、阈值后掩码、质心坐标、峰值内存、Flash 占用。

验收：模型成功加载；输出形状正确；PC 与板端差异在预先记录的容差内或有明确的量化/布局解释；连续运行不少于 100 次无崩溃和内存持续增长。

失败路径：加载失败时依次核对 ESP-DL/ESP-PPQ 格式版本、模型存放方式、PSRAM 配置和算子日志；固定图闭环通过前不接相机。

实机兼容性补充：当前最终候选图包含 4 个 `mode=linear`、`align_corners` 的 Resize。ESP-DL 3.3.1 在解析其空 ROI 可选输入时于 `get_operation_parameter` 触发 `LoadProhibited`；独立对齐模型分区复测后崩溃位置不变，已排除存放方式与地址对齐。继续推进前需二选一：训练并重新锁定最近邻上采样模型，或为 ESP-DL 实现并验证双线性 Resize；不得直接修改 ONNX 后沿用原精度结论。

最近邻路线补充：用户已选择最近邻重训。软件闭环已生成无空 ROI 的最近邻 `.espdl`，最终测试空预测率为 `0.6993%`，高于原候选的 `0%`。用户已明确允许该候选先做上板工程验收；此授权不改变其未通过原精度门槛的事实。详见[阶段 4：最近邻上采样重训与量化结果](阶段4_最近邻上采样重训与量化结果.md)。

### 4C：可见光 OV2640 采集链路

输入：OV2640 帧缓冲区。

处理：优先获取 YUV/RGB 帧并提取亮度，执行与电脑端一致的灰度化、等比例缩放、补边和输入量化；复用单一缓冲区，避免 JPEG 解码成为默认路径。

输出：原始帧尺寸、`64×64` 灰度输入摘要、取帧/预处理/推理/后处理耗时、中心坐标。

验收：连续采集与推理不少于 300 帧；无帧缓冲泄漏；串口分段报告平均值、P50、P95 和最大耗时。

失败路径：相机初始化失败时核对板型引脚和 XCLK；内存不足时先降低相机帧尺寸、缓冲数量并复用内存，再考虑修改模型。

### 4D：红外域验收

输入：后续获得的红外 OV2640 或等效红外眼图采集模组，以及独立标注的自采样本。

验收：硬件链路复用 4C；自采红外图只作为外部泛化测试，单独报告样本数、标注方式、光照与失败样本，不并入 LPW/HMEPS 主测试结论。

失败路径：没有红外模组时，阶段 4 可完成模型上板和可见光采集工程闭环，但项目必须继续标注“红外端到端精度待验证”。

## 服务层与表现层分工

- 相机服务：初始化 OV2640、取得和归还帧缓冲区，不负责模型推理。
- 预处理服务：把相机或固定图转换为模型要求的单通道 `64×64` 输入。
- 推理服务：加载 `.espdl`、执行 ESP-DL、暴露输出张量与耗时。
- 后处理服务：阈值化、最大连通区域与质心计算。
- 串口表现层：全中文输出状态、错误、坐标和性能统计，不承载业务计算。

## 输入、输出与失败状态

- 正常输入：固定测试图或相机帧。
- 正常输出：掩码摘要、瞳孔中心坐标、各阶段耗时、内存与固件占用。
- 可恢复失败：取帧超时、单帧无有效连通区域；记录并继续下一帧。
- 不可恢复失败：模型加载失败、输出形状不符、PSRAM 不可用；打印中文错误并停止推理循环。

## 建议人工操作

1. 提供开发板正反面清晰照片，确保模组丝印和接口可见；若有商品链接或原理图一并提供。
2. 用确认可传数据的 USB 线连接电脑，告知设备管理器中新出现的串口名称。
3. 首次烧录时按板卡说明操作 BOOT/RESET；固件运行后让板卡保持连接，以便自动读取日志。
4. 相机首次取帧时对准普通高对比物体，不直视强红外光源，也暂不以眼睛分割效果判断模型好坏。
